## 1. 專案目標

本專案目前的核心目標不是單純證明模型比臨床規則更準，而是建立一個能處理不完整聽力資料的 ear-level multi-task learning pipeline。

目前資料中會出現：

- missing value
- AC/BC NR
- ABG censored evidence
- tympanogram NP
- 左右耳 label / feature 需要拆開訓練

目前較合理的研究定位是：

```text
Clinically guided ear-level multi-task hearing classification
under missing / NR / censored audiological measurements.
```

論文主結果建議以 no-support deployment validation 為主，因為 dashboard 實際推論固定使用 `support=None`。configured validation 可保留為 development validation 或 ablation。

## 5. Task2 clinical rule

Task2 rule 目前集中於 `clinical_rules_v74.py`。

標準 label：

- `WNL`
- `SNHL`
- `CHL`
- `MHL`

已排除：

- `high tone loss`
- `low tone loss`
- `high and low tone loss`
- 空值或非標準 hearing type

核心判斷邏輯：

1. 只看 500/1000/2000/4000Hz。
2. 逐頻率判斷，不使用 PTA、`ac_mean`、`bc_mean`、`abg_mean` 作為 hearing type rule。
3. 若任一有效頻率有 `ABG > 10`，代表有 air-bone gap。
4. 有 ABG 時：
   - 任一有效 BC `> 25`：`MHL`
   - 否則：`CHL`
5. 沒有 ABG 時：
   - 任一有效 AC `> 25`：`SNHL`
   - 否則：`WNL`
6. BC missing 不會被當成 0，也不會產生 fake ABG。
7. 若 evidence 有 missing/NR/censored，reward confidence 會下降。

## 6. 模型與 loss

模型由 `mtl_meta_irl_transformer.py` 定義。

主要元件：

- NumericFeatureTokenizer
- TransformerEncoder
- task embedding
- FiLMAdapter
- classification heads
- reward heads
- task uncertainty `log_vars`
- prototype meta-learning support/query mechanism

每個 task 的 loss：

```text
task_loss = exp(-log_var_task) * CE_loss
            + log_var_task
            + reward_weight * reward_loss
```

其中：

- `CE_loss`：class-weighted CrossEntropyLoss。
- `reward_loss`：reward head 與 expert consistency target 的 MSE。
- `reward_confidence`：對 missing/NR/censored 或不完整 rule evidence 降低 reward MSE 權重。
- `reward_weight`：控制 reward loss 對總 loss 的影響。

## 12. 論文解讀注意事項

1. Task2 label 與 clinical rule 高度一致，因此不宜把主張寫成「模型已證明優於 clinical rule」。
2. 更合理的主軸是：
   - missing/NR/censored evidence 下的穩定性
   - no-support deployment performance
   - multi-task 是否輔助少樣本任務
   - reward confidence 如何避免 rule 過度主導 CE loss
3. CHL 只有 2 筆，Task2 macro-F1 必須搭配 per-class recall/F1 解讀。
4. Task3 B/C 樣本少，應搭配 subgroup/error analysis 檢查 NP/missing flags 的作用。
5. `full` 模式中的 support/prototype 屬於 configured validation / ablation；dashboard 實際使用時應看 no-support。
